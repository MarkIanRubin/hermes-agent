# Iterative Static SVG Site Edits

## References

- `references/businessvisualization-money-model-zoom-labels-2026-05-07.md` — Money Model zoom player placement, centered playback behavior, Revenue upper-left direct label placement, cache-busting, and browser geometry snippets.

## When to Use

Use this when the user asks for incremental visual changes to an existing static site or inline SVG asset, especially when they are pointing at a screenshot and saying things like:

- "move this label/button"
- "rename this heading"
- "line these words up"
- "remove this section/button/text"
- "convert these text markers into icons"
- "animate/color-code this SVG line"
- "replace this old PNG/diagram with a cleaner SVG version"
- "create a new diagram in this same style/pattern"
- "add a data-backed visualization/dashboard section to this static site"
- "look back at an older version and bring the best diagrams/copy/ideas into the current site"
- "turn this local database/export into a client-facing static visual model"
- "undo that" after a committed visual slice
- "commit it" or the repo workflow implies committing each slice

This is for editing an existing artifact, or creating a close sibling asset by pattern-matching an existing SVG's style, layout, and verification flow. For unrelated new architecture diagrams, use `architecture-diagram`; for brand-inspired page generation, use `popular-web-designs`. If the user asks for a 1-for-1 pattern match to an existing SVG, prefer cloning the proven asset and systematically replacing semantic labels/sections instead of generating from scratch.

## Workflow

1. **Identify the target precisely**
   - If the user provides a screenshot, inspect it when vision tools work.
   - If vision fails and the screenshot contains text, run OCR on the cached image before guessing, for example `tesseract /path/to/image.jpg stdout --psm 6`, then search the repo for the recovered string.
   - If OCR is partial, use the recovered distinctive words plus repo search and browser snapshots to infer the obvious target instead of stalling.
   - When a user asks to restore or integrate the "best old stuff," inspect git history before inventing new content: use `git log --oneline --decorate --all`, `git show <commit>:path`, and targeted greps for the remembered concepts. Treat old copy/diagrams as source material, then synthesize the strongest ideas into the current layout rather than blindly reverting the page.
   - If old JS/data already contains a richer workflow than the visible HTML exposes, prefer wiring the existing data into visible controls/cards before creating duplicate content. Verify counts match the backing data (for example all steps in a JS array have corresponding buttons/nodes in the DOM).
   - Search for exact visible strings/classes in HTML/CSS/SVG.
   - When a task involves similarly named products/repos, or the user corrects the project identity, write a quick scope contract before editing: target domain/repo/product name, explicitly excluded repo/product, and the public name that should appear in the UI. Then work only in that target repo, update durable memory if the distinction will recur, and verify the old ambiguous name is gone from rendered pages unless preserved only in redirects/legacy URLs.
   - When renaming a public tool/product in a static site, update page titles, meta descriptions, nav labels, anchors, aria labels, prompts/copy blocks, JS-generated text, cache-busting query strings, and old-route redirects. Verify both source search and rendered DOM text so stale generated copy does not survive.
   - When a word appears in both labels and descriptive copy, target the exact SVG/DOM text node the user indicated; don't globally replace unless explicitly asked.

2. **Make surgical edits**
   - Prefer editing the smallest relevant asset: usually the `.svg`, `.html`, or `.css` file.
   - When creating a sibling diagram in the same visual language, start from the existing SVG source (or a prior committed version via `git show <commit>:path`) and replace labels, modules, and metaphors surgically. This preserves spacing, typography, gradients, shadows, and verification habits better than hand-building a fresh file.
   - Preserve existing style vocabulary, spacing system, colors, and animation patterns.
   - For SVG alignment, match text/icon `x`/`y` coordinates directly when the user says items should line up.
   - When aligning generated diagram elements to a semantic axis/timeline label, do not eyeball from screenshots. Read or compute the reference label coordinate from the generator (`timelineStart + span * index / (n - 1)` or the rendered `.machine-timeline-label[x]`), reuse that coordinate for related gears/callouts, and verify with browser `getBBox()`/`data-*` centers that deltas are near zero. Keep adjacent categories separated by their own axis labels (for example expenses under `EXPENSES`, profit under `PROFIT`) instead of letting a column drift between them.
   - When the user requests a physical movement such as “move this label 2 cm left,” convert the physical distance through the live rendered SVG scale instead of guessing: `cssPx = cm * 96 / 2.54`, `svgUnits = cssPx / (svg.getBoundingClientRect().width / svg.viewBox.baseVal.width)`. Apply the coordinate delta, cache-bust the consumer, then verify in the browser by comparing target and reference `getBBox()` centers and reporting the converted cm.
   - When moving one cog/gear in a generated SVG, move the generator's gear coordinates and any explicit label coordinates together. Prefer changing the gear object/slot source (for example `y` and `labelY`) so derived chains, mesh lines, packets, and labels recalculate from the new center instead of patching rendered DOM positions individually; then verify the target gear `cy`/bbox, label position, connected path existence, removed text if applicable, cache-busted script URL, console errors, and overflow.
   - For SVG node/flow diagrams, keep connector paths visually behind opaque node/card fills so lines or animated tokens do not run through words. If the user says connectors should only connect boxes, remove redundant interior/faint connector paths, reroute the semantic flow path between node edges, move nodes outward for breathing room, and draw moving tokens before nodes so boxes mask them.
   - For animated flow icons, every `<animateMotion>` must reference the visible semantic path it is meant to follow (`<mpath href="#...">`). Remove stale hidden loop paths/groups that create icons drifting across unrelated areas, and add small explicit token groups only for flows the user asked to see.
   - When a user points at a stray SVG artifact line in a screenshot, do not stop after removing the first plausible connector — and do not remove semantic paths just because they share the same color. In the live browser, inventory all `path`/`line` elements with class, `d`/endpoints, computed `stroke`, `strokeWidth`, and `getBBox()`; filter for lines crossing the complained-about region/color, then classify each candidate as either intended semantics (timeline, flow chain, labels/tokens) or artifact (stale radial/guide/connector). Preserve or restore intended systems and remove only the artifact source, usually by excluding that gear/connector case at render time. Verify both sides: intended selector counts remain positive (for example timeline/flow path/tokens/labels still exist) and artifact selectors/geometry in the target region are zero.
   - When the user draws a bright annotation line over an artifact and vision tooling fails, use the image file directly instead of guessing. Inspect metadata (`file`, `sips`), and if needed install/use Pillow to threshold the annotation color (for example high-G green pixels), compute its screenshot bbox/slope, then map screenshot coordinates to the live SVG `viewBox` using the SVG `getBoundingClientRect()`. Use that mapped line to score nearby SVG elements by `getBBox()` distance/class/stroke/fill, then remove the real generator/CSS for the artifact. Verify removal by selector counts for the artifact classes (for example stale chain paths/links/packets = 0), not by visual vibes.
   - If the artifact is still ambiguous after browser inventory and screenshots/vision fail, add a temporary **trace mode** rather than guessing. Gate it behind a query param such as `?trace=1`, convert pointer `clientX/clientY` into SVG coordinates with `createSVGPoint().matrixTransform(svg.getScreenCTM().inverse())`, draw the user's drag as a bright polyline, then on `pointerup` score candidate `path,line,circle,polygon,polyline` elements by how many traced points fall inside their `getBBox()` plus padding. Show a high-z-index, obvious panel with the top candidate classes/IDs/strokes/paths and draw red/gold hit boxes around matches. Make release robust with `pointercancel` and a `window.pointerup` fallback, update the panel while dragging so the user knows it is working, cache-bust the JS URL, and remove the trace tool after the artifact is identified/fixed.
   - When the user asks for specific SVG icons to flow, reuse existing `<symbol>` icons via `<use href="#icon-name">` inside the animated token instead of generic text markers.
   - When replacing marketing/channel tokens with researched channel attribution, update both the animated packet icons/labels and the semantic backing model used by any upstream gear/wedge. Keep the attribution as an explicit channel-mix constant or data field with `source`/assumption notes, use stable `data-*` attributes such as `data-segment-job-share`, preserve the existing motion path/topology, cache-bust CSS/JS, and verify browser-visible text, packet icon labels, wedge shares, source metadata, console errors, and overflow.
   - When animated icons must pass through a visible sub-loop/figure-8 before returning, put that loop geometry into the same `id` path referenced by `<mpath>`; drawing a separate decorative loop is not enough because the icons will bypass it.
   - For connector paths that should start/end on an outer box, compute/use the absolute edge coordinate of the source/target card (`group translate + rect width/height`) and move the SVG path `M`/end point there, not over labels or card interiors.
   - When repeated visual edits change the meaning of a diagram, update surrounding explanatory copy, `<desc>`, and `aria-label`s so they describe the current model instead of stale earlier language.
   - When a user asks to rename a visible heading/section in a generated inline SVG (for example “X P&L” → “Income and Expense”) and says “whatever is after that,” treat it as a visible semantic label change, not a global data-source rename. Update the generator text, nearby UI control labels/ARIA (zoom buttons, container labels, SVG `aria-label`), and cache-busting query strings together; preserve source-data/workbook references unless explicitly asked to sanitize or relabel them.
   - When the critique is "make the fonts easy to read" or similar, improve legibility at the SVG source level: raise label/value/status font sizes, increase font weight, add `paint-order:stroke` with a subtle stroke/halo behind text when labels sit on dark or busy shapes, and verify the direct SVG accessible text still reads cleanly. If the user later says they still do not see the change, assume the prior edit was too subtle until proven otherwise: make the delta unmistakable (for example 20–40%+ larger text, thicker halos, larger key nodes) rather than inching values upward.
   - When a generated SVG gear/process diagram has bulky label cards that make the machine feel like a dashboard, move the primary semantic labels and compact values into direct in-SVG text below the corresponding gear centers instead of preserving separate card groups. Suppress/remove the old card renderer for those direct-labeled gears, keep values compact (`$88.8M` rather than full currency if space is tight), add a shared direct-label class for styling, cache-bust JS/CSS consumers, and verify: old card selector count is zero for the targeted gears, direct label/value counts match, each label/value `x` or `getBBox()` center equals its gear center, the requested labels share one horizontal `y` when asked, accessible SVG text still contains the values, console errors are zero, and page overflow is zero.
   - When the user asks to reuse an existing direct gear-label style for a different gear, copy the existing class/style contract instead of inventing a new card. Derive the value from the target gear/data source, use compact formatting consistent with the reference label, suppress any old card label for that gear if the goal is the clean direct style, cache-bust the consumer, and verify with the target gear's actual visible geometry. If the gear group bbox includes the newly added label, compare against the teeth/face bbox (for example `.machine-teeth` or `.machine-gear-face`) to prove the label sits above/centered on the gear.
   - When a screenshot points to text inside a multiplier/driver gear and the user wants it removed, remove only the visible overlay text selectors while preserving the gear and its `<title>`/semantic data. If the replacement label should sit above a related top/idler gear, reuse the existing direct-label classes, compute derived business value from the connected gear/data source (for example marketing spend × ROAS), cache-bust the script, and verify: old in-gear text selectors are zero, new label/value are above the target gear by `getBBox()`, value math matches the source data, console errors are zero, and overflow is zero.
   - When showing multiple transactions or particles flowing through a system, prefer one shared semantic path with several token groups using staggered `begin` offsets and `<mpath href="#pathId">`, rather than drawing separate near-duplicate paths. Verify token count, `animateMotion` count, and path reference count from the fetched SVG source. For “I don’t see the transactions” feedback, increase both count and visual salience: larger circles/icons, brighter path/stroke opacity, shorter animation duration if appropriate, and staggered offsets that distribute tokens across the whole path at initial load.
  - When the user asks animated timeline/flow icons to represent recent real business values, do not leave decorative symbols or hand-enter constants in the JS. Source the latest valid actual records from the backing data artifact/export, add a small repeatable JSON field such as `weeklyRevenueSamples` with date/value/status/source metadata, render each token with semantic `data-*` attributes plus readable value text (for example a `$` hexagon with compact `$19.96K` above it), cache-bust CSS/JS, and verify JSON validity, token/label/icon counts, sample values, console errors, and visual readability in the browser.
  - When the user asks to change animated transaction/timeline numbers to typical price-list or job-size examples, update the backing JSON samples and any JS fallback defaults together instead of only editing visible SVG text. Choose values that satisfy the requested distribution/range, update `source` metadata to identify them as modeled price-list/job-size examples rather than daily actual revenue, preserve semantic `data-sample-value` attributes, cache-bust the consuming HTML/JS, then verify JSON validity, JS syntax, browser-visible SVG text, `data-*` sample values, and console errors.
   - When changing the visual *style* of animated packets/tokens (for example symbols → gear tokens, colored dots, or segment icons), preserve the existing semantic chain/path topology unless the user explicitly asks to remove it. It is easy to overcorrect and accidentally rip out the visible chain while swapping token primitives. Verify both the unchanged infrastructure and the new token style: chain path/ring/link counts, every `<mpath href="#...">`, packet/token counts by flow family, segment/icon labels, console errors, and overflow.
   - When the user asks for a minimum percentage of chain nodes to carry icons/labels, treat it as an explicit coverage target against the chain's moving-link/node count, not a vibe. Compute `iconNodeCount = Math.ceil(chainNodeCount * targetCoverage)`, stagger the icons across the same semantic `<mpath>`, add readable compact labels above the token (for example `CPM $38`), cache-bust CSS/JS, and verify in the browser: total chain nodes, icon-token count, `coverage >= target`, label distribution/counts, `<mpath>` hrefs, console errors, and visual clutter/readability.
   - When the user asks for a single verb/title that summarizes a group of labels, treat it as a semantic umbrella edit: infer the verb from the child labels, add/replace the title and icon together, preserve the child labels, update nearby explanatory copy and `<desc>`, and verify the old title string is gone.
   - When the user asks to add visual icons to existing SVG labels, preserve the original words as the stable semantic content and add icons as a visual layer. For quick static-site prototypes, emoji can work if you set an explicit emoji font stack (for example `.icon{font-family:"Apple Color Emoji","Segoe UI Emoji","Noto Color Emoji",sans-serif}`), keep icon text separate from label text, and verify the SVG still renders on the target machine.
    - For removals, remove the visible element first; only remove supporting CSS/JS if it is clearly dead and in scope.
   - When replacing a raster diagram (`.png`, `.jpg`) with a new SVG, update the consuming HTML/card to point at the SVG, keep the old asset out of the page, preserve the site's existing palette/typography, and verify the browser no longer requests or references the old raster file.
   - When replacing an old SVG/visual that appears in multiple places, search all source references and update every consumer in scope (hero image, artifact/gallery card, modal source, captions/alt text, and cache-busting query strings). Do not stop after the first visible occurrence. Verify with source search plus live browser image inventory (`browser_get_images` or `[...document.images].map(img => img.src)`) that the old asset is no longer loaded and the new asset is loaded everywhere expected.
   - When the user asks for **one version** of a visual/model across multiple static pages, choose a single canonical implementation instead of maintaining duplicate static captures. If a dynamic page already renders the right model, embed or route to that dynamic source (for example an iframe with an `embed` query mode), remove the old static SVG/PNG asset and any dead expand/modal code tied to it, and verify there are zero old asset references in source and zero old image requests at runtime. For embedded canonical models, verify the iframe document has the expected semantic labels/colors, no internal scroll (`scrollHeight <= clientHeight`, `scrollWidth <= clientWidth`), and the host page has no overflow.
   - When editing an asset already referenced with `?v=` or another cache-busting parameter, update every consuming URL's cache-buster as part of the same change. A correct SVG edit can look unchanged to the user if the page still loads the old cached URL; verify the live `document.images` URLs include the new cache-buster.
   - When the user asks to move an image/card stack below nearby copy or a CTA link, first check whether the layout is controlled by CSS grid/flex rather than DOM order. Often the smallest safe edit is changing the parent from a multi-column grid to one column, adding a max-width to the copy block, and preserving the existing image/card markup.
   - When a screenshot shows a simple ordered link/list of model names but repo search does not find the exact strings, inspect the rendered page and nearby captions/alt text. The visible list may need both a navigation/link entry and a backing card/figure/artifact in the same order; add the item in both places when that matches the page pattern, then verify with DOM text and relative geometry.
  - When fixing a crowded static-site header/nav, identify whether the problem is caused by flex distribution, long pill labels, missing `white-space: nowrap`, stale cache-busted CSS, or breakpoint order. Prefer tightening the existing flex contract (`gap`, `margin-left:auto`, `flex:0 0 auto`, `overflow:hidden`) and shortening overlong labels before inventing a new menu system. On tablet/mobile, hiding secondary nav links is often cleaner than letting them wrap into a bad pill stack.
  - When the user asks menu options to "all look the same," remove special-case nav classes/styles from individual links where possible and put all visible nav links under the same `.nav-links a` styling contract. If a page has CTA pills outside the main nav group, move them into the same group or deliberately exclude them. Verify uniformity with computed styles and bounding boxes, not just eyeballing: text list, removed labels, height, padding, border radius, font weight, active state, and horizontal overflow.
   - When adding a data-backed visualization to a static site, keep the data generation repeatable: create a small exporter script under `scripts/`, write a static JSON artifact under `data/`, and have the frontend `fetch()` that artifact. Do not hand-copy numbers into HTML/JS if they come from a DB or workbook.
   - When the source is a local `.xlsx` workbook and spreadsheet libraries are unavailable, parse the workbook directly as ZIP/XML with Python stdlib: read `xl/workbook.xml` + `xl/_rels/workbook.xml.rels` to map sheet names to worksheet files, parse `xl/sharedStrings.xml`, then extract cached cell values from the target sheet. Add a small regression test with a generated minimal workbook so future extractor changes prove the row/column mapping still works.
   - For P&L/income-statement visualizations, extract source rows by stable row labels (for example revenue, total expense/money out, net income/profit), compute conservation checks from the exported JSON (`Revenue - Money Out - Profit == 0`), include percent-of-revenue fields for rendered labels, and verify both JSON math and browser-visible metrics.
   - When the initial P&L export only has rollups but the user expects “details in income and expenses,” inspect the workbook itself before asking for another file. Enumerate sheets and text-rich tabs, then parse the full P&L/account-level sheet (often named like `2026 Plan - P&L`) by section headers (`REVENUE`, `Cost of Goods Sold`, `Expense`, `Other Income`, `Other Expense`). Preserve account-code rows, parent subtotal rows, total column values, and monthly columns (for example Total in `L`, months in `N:Y` for GJ planning workbooks). Verify hierarchy math, not just top-line conservation: Revenue − COGS = Gross Profit; Gross Profit − SG&A = Net Ordinary Income; Net Ordinary Income + Other Income − Other Expense = Net Income. Keep rollup gears as a simplified view only if the detailed account tree is not yet wired into the visual.
   - For budget-backed public business visualizations, distinguish **budget data**, **actual observed data**, and **model placeholders/pending source data** in both the JSON and rendered UI. If you derive proportional visuals from incomplete budget data (for example funnel volumes or profit/expense splits are missing), label those assumptions explicitly and avoid presenting placeholders as actuals.
   - Before publishing exported operational data into a public/static site, strip or generalize client/location/source-machine details that are not part of the public artifact, while preserving enough source notes to explain whether numbers are budget, actual, or pending.
   - For operational/business dashboards, separate actual observed data from future scheduled/goal-only data. Apply a current-date or latest-valid-snapshot cutoff so future zero-revenue days do not pollute actual-vs-goal ratios.
   - For static data visualizations, add semantic DOM text for the key numbers and labels, not just canvas/SVG pixels, so browser snapshots can verify the result and screen readers can read it.
   - When the user's critique is about the *visual metaphor* (for example "this should be a gear train, not rows/cards"), change the underlying visual structure to embody the metaphor instead of cosmetically restyling the old layout. Preserve data semantics, but make primitives, spatial relationships, motion, labels, and sizing all reinforce the requested metaphor; then verify both meaning and mechanics (for example spin direction/speed, proportional sizes, connected layout, console errors, and overflow).
   - For data-backed machine/energy-transfer visualizations, replace floating cards with a canonical SVG generator rather than manually positioning styled divs. Encode the machine semantics directly: proportional gear radius, tooth counts, meshed spacing/ratio labels, shafts, pitch/construction circles, leader lines, alternating rotation directions/speeds, reservoirs/chambers, flywheels/cranks, animated packets on semantic `<path>` elements, and conservation-law labels. When business math defines a mechanical relationship (for example ROAS, conversion rate, margin, near-equal stage deltas, or speed reduction), encode that relationship in both the data/copy and the rendered mechanics: gear value, gear size ratio, animation duration/speed ratio, intermediate reducer gears, labels, and assumptions. When adding an intermediate idler/reverser gear, preserve the intended ratio explicitly: for equal-radius 1:1 reversal, match radius and animation duration; for unequal-radius gears, preserve rim/tangential velocity instead with `idlerDuration = sourceDuration * (idlerRadius / sourceRadius)` and `driverDuration = sourceDuration * (driverRadius / sourceRadius)`, then verify `(radius / duration)` ratios against the source in the browser. Flip the animation direction, compute visible rim contact from actual rendered gear geometry, reroute any belt/chain to the idler instead of the original gear if that is the mechanical connection, and verify old chain/ring selectors are zero while packet `<mpath>` refs still point to the active semantic chain. For small/fast idlers, use a semantic label like `REVERSER` and keep them outside value-proportional P&L scale unless the user explicitly says the idler represents dollars. For tiny deltas (for example 0.25% larger), avoid eyeballing: set the JSON/exported values and rendered radius/duration from the same ratio, then verify the value ratio, face-radius ratio, and inverse animation-duration ratio numerically in the browser. If two adjacent gears look identical but must read as different, increase compact-label precision only enough to expose the difference (for example `$7.32M` vs `$7.34M`), add `data-*` attributes for source values/radii on the semantic connector or gears, and verify both visible labels and numeric metadata show the intended difference without changing scale math. When a user says gear values must be proportional in **size** to a baseline like Revenue, use visual area as the proportional quantity unless they explicitly ask for radius/diameter: `radius = baselineRadius * sqrt(value / baselineValue)`. When speed/velocity must be proportional to value, animation duration is inverse: `duration = baselineDuration / (value / baselineValue)`. Verify every non-exempt gear with browser-computed `areaRatio = (radius / baselineRadius) ** 2` and `velocityRatio = baselineDuration / duration`, compare both against `value / baselineValue`, and report max deltas. If a correctly proportional slow gear looks static or wrong (for example profit at ~14% of revenue speed), do not distort the math to make it feel faster; expose the computed duration/ratio in `data-*` attributes or label text, add a small visual witness mark/dot on the rotor so slow rotation is perceptible, and verify `computedDuration`, `valueRatio`, and `speedRatio` in the browser. If the user says a spin witness/bar is offset from the hub, replace one-sided marks with symmetric geometry centered on the gear center (for example two equal-length `<line>` arms forming a `+` whose endpoint averages equal `gear.x`/`gear.y`), and set the animated group's rotation origin to that exact center. For absolute-coordinate SVG children inside their own helper rotor, use `transformOrigin = `${gear.x}px ${gear.y}px` with `transformBox = 'view-box'`, or alternatively use a translated local group with `transform-origin: center`; verify line endpoints, transform origin, animation name, console errors, and visual centering in the browser. When adding multiline SVG labels with `<tspan>`, remember accessible text can concatenate adjacent tspans; include an explicit space/trailing space or verify the browser snapshot reads cleanly. When the user says to make multiple panels/diagrams into **one diagram**, remove the visual bridge/center connector and the duplicate panel DOM, then render a single wider canonical SVG in the dominant requested style; preserve semantic sub-sections with in-SVG section labels rather than separate page panels. For joined business-machine diagrams, align the shared handoff axis numerically (for example Customers and Revenue gear centers on the same `cy`) and verify the connector is a continuous horizontal mesh line so it reads as one machine, not two glued-together halves. For P&L machine diagrams, prefer exposing detailed source rows (COGS, SG&A, Other Expense, Profit) as individual gear items sourced from the budget/export JSON, update the conservation equation to match those row-level gears, and keep aggregate Money Out only as a metric unless the user asks for both. When the user asks for the “top N most valuable expenses,” rank leaf expense rows by absolute/total value first, preserve the original rank number after grouping, exclude subtotal/percentage/result rows, and group by business family such as Related COGS vs Related Expenses plus named buckets derived from section/header and account keywords. Render those family groups as explicit in-SVG headings/columns, keep bucket names in each label, and verify the grouped view by asserting the expected detail gear/label count, family headings, bucket counts, zero label overlaps via SVG `getBBox()`, zero out-of-view labels in both full and zoomed viewBoxes, console errors, and horizontal overflow. If the detail view gets too crowded and the user asks for a smaller top N, keep only the requested top N as individual proportional gears and aggregate every remaining leaf expense row into an **Other Expenses** pie/donut chart; include the remaining-row count, bucket-level pie slices, and source values in JSON so the frontend stays repeatable. For UI-scaled expense gears, make the top-N gear **area** proportional to expense value within the intended comparison set (for example `radius = maxUiRadius * sqrt(value / maxTopValue)` only when the visual is explicitly a local top-N expense comparison). If the user compares an expense to Revenue/Profit or says there is a scale problem, switch the comparison set back to the global business baseline: `radius = revenueRadius * sqrt(value / revenueValue)` for **all** value gears in that comparison, so Wages at 12.4% of Revenue is slightly smaller than Profit at 14.1%. Then choose the speed law deliberately and verify it, because business-value speed and meshed-gear speed are different metaphors: if the diagram is communicating value velocity, use `duration = revenueDuration / (value / revenueValue)` and verify `speedRatio = revenueDuration / itemDuration`; if the diagram is communicating a physically meshed gear train or the user says a smaller gear such as Profit is not spinning fast enough relative to Revenue, preserve rim/tangential velocity with `duration = revenueDuration * (radius / revenueRadius)` and verify `radius / duration` matches Revenue. Do not mix local max-based expense speed with revenue-based profit speed or value-speed with meshed-speed in the same mechanical comparison unless explicitly labeled. Verify `valueRatio`, area/radius ratio, and whichever speed law was chosen in the browser; do not use an additive radius offset if exact proportionality matters. When rendering an aggregate Other Expenses pie in a gear-machine metaphor, keep it visually in the same mechanical language as the rest: draw gear teeth with `gearPath`, put category wedges inside the gear face, spin the rotor like other expense gears, and verify teeth, rotor, wedge count, and distinct category colors in the DOM. If the user constrains the palette (for example only red/orange/green or an old-master/Da Vinci earth palette), centralize the palette map, map aliases like `gold` to an allowed color, set explicit `fill` on pie face/teeth/slices, and verify computed fills for gear faces, teeth, and pie slices contain only allowed RGB values. When the user wants the P&L to read as a machine rather than a list, make Revenue/Income the central hub gear, orbit detailed Expense gears around it in a radial layout, spin expense gears clockwise when modeling money-out, place Profit as a smaller output gear on the right of the hub when possible, and use staggered outside labels with leader lines instead of label text crowded inside/near meshing gears. If the user specifies a clock-face layout (for example Marketing at 12 o'clock, next largest expenses at 12:30/1:00/etc., Profit at 3 o'clock), treat those clock anchors as semantic constraints: compute every orbit point from the hub center and a shared orbit radius (often the existing Marketing radius), sort expense gears by value before assigning clock slots, reserve Profit at the exact 3 o'clock point even if another expense would land there, and use direct persistent labels/value/hourly text when the user says “like Marketing” rather than tooltip/callout animation. If the user says the chain should look like the **right half of a circle**, distribute the chain on the right semicircle from 12 o'clock through 6 o'clock (Marketing can be the 12 o'clock anchor and Profit the 6 o'clock endpoint), include the fixed anchor gear in spacing calculations, and derive the orbit radius from adjacent gear radii plus requested white-space padding instead of eyeballing. ## Pitfalls

- Screenshot vision may fail in Telegram/Codex sessions; don't let that block obvious text-based edits.
- Browser `innerText` will not include text inside an SVG loaded through `<img>`; fetch the SVG file directly instead.
- Static PNG render confirms parse/render, but not animated color timing.
- Removing a navigation button may leave another identical hero button; search all matching text before declaring done.
- User may say "remove this" after a prior removal. Treat it as the next visible matching thing and verify by browser snapshot.
